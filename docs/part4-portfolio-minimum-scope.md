*(카카오, 라인, 쿠팡, 배달의민족, 토스, 네이버 등 공고 분석)*

|------|-----------|-------------|

| TypeScript strict 모드 | ★★★★★ | ✅ 적용됨 |

| 다크모드 구현 | ★★★☆☆ | ✅ 구현됨 |

### 실제 채용 합격 포트폴리오 공통 특징
1. **기능이 실제로 동작한다** — 데모 링크 클릭 시 바로 확인 가능
2. **README가 충실하다** — 기능 소개, 스크린샷, 기술적 의사결정 설명
3. **한 가지 기능이라도 깊이 있다** — 얕은 기능 10개보다 깊은 기능 2~3개
4. **코드 품질이 일관적이다** — TypeScript 에러 없음, ESLint 통과
5. **배포 URL이 있다** — 면접관이 즉시 확인 가능해야 함

## 포트폴리오 최소 요건 (Must Have)

### 1. 실제 동작하는 배포 URL
- **필수도: 최우선**
- Vercel 무료 플랜으로 충분
- `.env` 설정, 도메인 커스텀 불필요
- **없으면 서류 통과율 급락**

### 2. 인증 흐름 완성 (로그인 → 대시보드)
- 현재 next-auth 미들웨어는 있으나 실제 로그인 UI가 없음
- **최소 구현**: 이메일/비밀번호 로그인 + 세션 유지 + 로그아웃
- 면접에서 "어떻게 인증을 구현했나요?" 질문이 반드시 나옴
- Mock 사용자 계정으로도 가능 (실제 DB 없이도 됨)

### 3. 대시보드 홈 — 현재 거의 완성 상태
- KPI 카드 ✅, 차트 ✅, 재고 ✅, 주문 테이블 ✅
- **추가 필요**: 실제 데이터처럼 보이는 스토리 (데이터 출처 설명)
- Mock 데이터지만 현실적인 수치와 변화 패턴 적용 필요

### 4. 주문 관리 페이지 완성 (PR #8 머지)
- 현재 PR OPEN 상태 — 머지만 해도 완성도 크게 향상
- 검색/필터/URL 상태 유지가 기술적 깊이를 보여주는 핵심 부분

### 5. Zustand 상태관리 실제 적용
- 현재 authStore, salesStore, uiStore 모두 비어있음
- 최소 **authStore** 하나라도 실제 구현해야 상태관리 경험 어필 가능
- 코드에 `import { useAuthStore } from '@/store/authStore'` 패턴이 있어야 면접에서 설명 가능

### 6. 유틸 함수 구현
- `formatCurrency`, `formatDate`, `useDebounce` 등 비어있는 파일들
- 30분이면 모두 구현 가능 — 비어있으면 감점 요소

### 7. README.md 충실화
- 현재 기본 README만 있음
- **필수 포함 내용**:
  - 프로젝트 소개 (한 줄 요약)
  - 주요 기능 목록 + 스크린샷/GIF
  - 기술 스택 및 선택 이유
  - 실행 방법
  - 배포 URL
  - 기술적 고민 및 의사결정 내용 (2~3가지)

## 포트폴리오 권장 추가 사항 (Should Have)

### 8. 제품 관리 페이지 (최소 CRUD)
- 주문 관리와 패턴이 동일 → Claude Code로 빠르게 생성 가능
- "관리자 대시보드"를 포트폴리오로 쓰려면 제품 관리가 없으면 설득력 부족

### 9. 단위 테스트 커버리지 확보
- 현재 3개 테스트 파일 존재 — 긍정적
- **권장 목표**: 핵심 유틸 + 훅 + 주요 컴포넌트 커버리지 60% 이상
- 테스트 코드 자체가 "협업 가능한 개발자"를 증명

### 10. 성능 최적화 1가지 이상
- 차트 dynamic import (가장 쉽고 임팩트 큼)
- Lighthouse 결과 스크린샷을 README에 첨부 → 차별화 포인트

### 11. 다크모드 완성도
- `useThemeToggle` 훅은 있음
- Header에 토글 버튼이 동작하는지 확인 필요
- 모든 컴포넌트에서 `dark:` 클래스가 일관되게 적용되었는지 검토

## 포트폴리오 차별화 전략 (경쟁 우위 요소)

현재 SellOps가 이미 갖춘 차별화 포인트:

### 강점
1. **한국어 초성 검색** (`choseong.ts`) — 국내 서비스 실무 경험 어필
2. **IME 처리** — 한글 입력 UX 세부 처리, 실무 경험 없으면 모르는 내용
3. **URL query 기반 상태 유지** — 공유 가능한 URL, 브라우저 뒤로가기 지원
4. **CodeRabbit 자동 코드 리뷰** — 협업 도구 활용 경험
5. **Recharts 커스텀 차트** — 단순 라이브러리 사용 이상의 커스터마이징
6. **ErrorBoundary** — 에러 처리 전략에 대한 이해

### 면접에서 설명할 수 있는 기술적 의사결정 포인트
1. "styled-components를 TailwindCSS로 전환한 이유" — 실제 마이그레이션 경험
2. "IME 처리를 별도로 구현한 이유" — 한국어 UX 이해
3. "URL query로 필터 상태를 관리한 이유" — 사용자 경험(링크 공유, 뒤로가기)
4. "Mock 데이터에서 실제 API 연동으로 전환할 때의 전략" — 점진적 교체 설계
5. "차트에 dynamic import를 적용한 이유" — 번들 크기, TTI 최적화

## 현재 프로젝트의 솔직한 평가

### 잘 된 것
- 대시보드 홈 화면 완성도가 매우 높음 (시각적으로 임팩트 있음)
- 코드 컨벤션, 폴더 구조가 체계적
- PR/이슈 흐름이 실무 프로세스를 따름
- 한국어 검색 등 세부 UX 처리가 차별화 포인트

### 아직 부족한 것
- 실제 동작하는 인증 흐름 없음 (면접관이 로그인을 못 함)
- 배포 URL이 없음 (화면 확인 불가)
- Zustand 상태관리가 실제로 사용되지 않음
- 비어있는 파일이 너무 많음 (완성도 인식에 부정적)
- 실제 API 연동 없이 Mock만 사용

### 결론
현재 상태에서 **약 10시간 추가 작업**으로 포트폴리오로 제출 가능한 수준에 도달 가능합니다.  
대시보드 홈 화면의 완성도가 높으므로 **인증 + 배포 + README** 세 가지가 핵심입니다.
